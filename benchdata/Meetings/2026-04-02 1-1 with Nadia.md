# 1-1 with Nadia – Queue Design for Campaign Processing

## Context

We talked through the campaign processing pipeline and whether CowQM is still the right choice as the queue layer. Nadia's main question was whether we're over-engineering the job structure or whether the current setup actually matches how campaigns run in production.

## Current Approach

CowQM sits in front of all the background jobs that prepare, fan out, and finalize campaign work. Redis handles job state, scheduling, and worker coordination. The job types we're running: campaign validation, audience expansion, message rendering, and delivery bookkeeping. Queue structure keeps high-volume campaign execution separate from lower-priority maintenance jobs so one noisy campaign doesn't block everything else.

## Reliability

We went through retries, backoff policy, and idempotency. Retries need to be explicit per queue — transient Redis or downstream API failures shouldn't cause duplicate sends. We also want proper dead-letter handling and enough queue visibility to catch problems without digging through logs. Worker concurrency needs a ceiling so we don't overwhelm downstream APIs during large campaign fans.

## Decision

Keep CowQM and Redis. No changes to the core architecture for now. Nadia wants to revisit backoff configuration once we have more data on which job types are actually failing and why.
