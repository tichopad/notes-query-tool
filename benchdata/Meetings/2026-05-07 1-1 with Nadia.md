# 1-1 with Nadia – Campaign Queue Throughput

## Context

Reviewed the plan for moving campaign delivery into CowQM so outbound processing becomes a proper background job instead of happening inline. Nadia wants the design simple enough to reason about but realistic enough to handle bursty traffic and delayed delivery windows.

## Throughput and Concurrency

Main topic was worker concurrency. We agreed on keeping a low default concurrency for campaign jobs, then scaling workers horizontally during peak send windows rather than cranking one worker too high. Aggressive concurrency can make the queue look fast while creating bottlenecks downstream in template rendering and provider calls. CowQM workers should report throughput separately from overall campaign completion time so we can tell where the slowdown actually is.

## Scheduled Sends and Failure Handling

Also discussed scheduled sends for campaigns that need to start at a specific time. The queue needs clean support for delayed jobs so they don't enter processing until the send window opens. On failures: retries with bounded backoff, clear dead-letter behavior, and enough metadata to distinguish transient provider issues from malformed campaign payloads.

## Next Steps

Nail down the concurrency defaults and document the retry/backoff policy. Nadia will draft a short spec so we can review it before implementation starts.
